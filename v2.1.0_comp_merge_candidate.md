# v2.1.0_comp_merge_candidate

## What is Cutter Compensation

Cutter compensation applies a lateral offset to the programmed toolpath to account for the physical cutter edge diameter, ensuring the tool follows the intended "real" geometry. More here..(/link to Nist reference or similar).

- G41: tool-left compensation.
- G42: tool-right compensation.
- G40: compensation cancel / return to nominal path.

Conceptually, compensation transforms coordinates within gcode motion commands such as G1 X(2.123), by recognizing the direction of travel and then applying a perpendicular offset by the indended or "real" tool radius while preserving junction continuity at line or arc transitions.

To visualize the effect of compensation: Imagine there is a multilane highway representing the programmed toolpath. The center lanes in each side of the highway are the nominal path. When G41 or G42 is activated, the tool shifts to the left or right lane to accomodate the vehicles size (tool diameter) and not crash into the barriers on either side. (finished edges of the part) 

The direction of travel relative to the inside road barrier determines whether G41 or G42 is appropriate. If the vehicle is driving in the fast lane while in the EU we would call this climb cutting, and if it were driving in the fast lane while in the US we would call it conventional cutting. Respectively driving in the slow lane in the EU is conventional and in the US is climb cutting. keeping in mind cutter rotation direction is always clockwise kind of like a helical wind terbine was fastened to the top of your vehicle.  

The compensation logic needs to know which side of the programmed path the tool should be on, so it can apply the offset in the correct direction to the correct side of the finished edge of the part.


## Executive Summary

This pull request offers near production-ready cutter compensation support for Carvera Community Firmware v2.1.0c with validated on-machine behavior and a refactored architecture designed for bolt-on integration, maintainability and flexibiliy.

Primary intended outcomes:
- Reliable streaming of gcodes through G41/G42 compensation preprocessor pipeline in G17 XY workflows.
- Line and Arc-aware compensation for G1 X,Y and I/J-formatted G2/G3 motion.
- Unified perpendicular offsets and corner / junction handling by using a simple geometric A-B-C intersection model. 
- Mininimization of memory leaks by using simple, deterministic ring buffer pair and explicit flush behavior.
- Integrated tool-diameter measurement workflow support via M491.2 TOOL_DIA and a additional support for D word workflows where gcode or controller is explicitly controlling the diameter.  This is intended to leave plenty of room for the emergence of a tool table construct.


## How This PR Accomplishes Cutter Compensation

The implementation introduces a bounded preprocessor stage between G-code intake and motion execution:

1. Intake command into Robot runtime.
2. If compensation is inactive: execute via existing path.
3. If compensation is active: feed command into compensation preprocessor.
4. Preprocessor computes compensated output moves when sufficient geometric context exists.
5. Robot executes generated compensated moves in order.
6. On G40, buffered moves are flushed before compensation disables.

This design preserves existing executor behavior and localizes compensation complexity to a dedicated module.

## Architecture: Dual Buffer Design

### Why dual buffers

Compensation needs local lookahead context in order to determine the direction of travel. A single move cannot always be offset perpendicular to the path correctly without awareness of the adjacent geometry. The design therefore maintains two synchronized ring buffers with different dedicated purposes:

- Uncompensated ring: source-of-truth, unmodified programmed points/moves.
- Compensated ring: generated, execution-ready offset output.

### Operational model

- The uncompensated ring collects incoming motion context.
- Once enough context exists in (A-B-C) coordinates, offset geometry is then solved.
- Compensated output is produced and queued into the output ring.
- Robot drains the output ring through normal execution code.

### Determinism and load behavior

- Bounded memory footprint (fixed ring capacity).
- Deterministic flush semantics on G40.
- Streaming debug diagnostics maintain visibility into produced-vs-served behavior when defined.
- The preprocessor is single-writer driven by intake flow, avoiding race-prone ownership.

## Architecture: Junction and Corner Functionality

### The A-B-C geometric model

Corner logic is solved from three points:
- A: previous point.
- B: current junction point.
- C: next point.

Cases:
- Onset case (A equals B): establish initial perpendicular offset using outgoing tangent.
- True corner (A, B, C distinct): solve intersection of incoming/outgoing offset lines to produce miter-consistent corner point.
- Terminal case (B equals C): create terminal perpendicular offset for clean exit.

### Why this matters

Earlier multi-path logic created branch-specific edge behavior and occasional diagonal artifacts at transitions. The unified A-B-C approach enforces one geometric rule-set for all junction classes, greatly reducing special-case drift and simplifying reasoning.

## Arc Compensation Strategy

### Supported form

- G2/G3 with I/J center specification in G17 XY plane.

### Implementation principles

- Compute effective offset radius from uncompensated radius and tool radius with side-aware sign.
- Preserve endpoint consistency to prevent snap artifacts.
- Use incoming tangent handling for terminal arc stability.

### Guard rails

- Reject unsupported patterns (notably R-format under active compensation).
- Maintain explicit error behavior rather than silent geometric fallback.

## TOOL_DIA Integration and M491.2 Certification

This branch supports non D word compensation workflows when TOOL_DIA has been measured and certified through M491.2 physical TDO process with the 3d toolsetter.

Key behavior:
- Manual TOOL_DIA writes are tracked as non-certified.
- M491.2 path stores certification marker in EEPROM reserve field.
- Non "D" G41/G42 only proceeds when certified diameter source is valid.

This ensures compensation diameter provenance and reduces operator risk from stale manual values.

## Probe and Spindle Safety Refinements Included In This Branch

### Side-probe dwell and retract behavior

- Second touch probing was modified to minimize sensor contact dwell while the bit is striking the 3d toolsetter sensor disk.
- Calibrate-pin handling in M491.2 mode now stops motion immediately on debounced hit rather than overtraveling to a safety margin threshold.
- Sequence ordering ensures retract completes before finalize and spindle stop.

### 3D probe spring-preload learning and toolsetter interaction

- Testing established that using the same spring behavior in both the 3D tool setter and the 3D probe body makes trigger order ambiguous during probe-tool TLO measurement.
- Increasing spring force in the 3D probe body makes the tool setter trigger first and prevents the tool setter from bottoming out like the stock carvera tool length sensor.
- That mechanical change improves trigger ordering but also means the probe body itself may never trigger during TLO measurement on the new tool setter.
- To support that hardware configuration without changing default behavior for other users, the firmware now provides an optional config-gated mode:
  - zprobe.use_3dtoolsetter true
- When enabled, probe-tool TLO calibration may stop on the tool setter's calibrate signal alone during TLO calibration, instead of requiring the probe body to trigger afterward.
- Default behavior remains unchanged for machines that still rely on the original dual-trigger sequence.

#### Why a static correction is currently required

- In the original dual-trigger model (tool setter signal followed by probe-body signal), firmware can observe both trigger events in one cycle and report their separation.
- On the 3D toolsetter path, there is no equivalent hard mechanical bottom-out event that guarantees the probe body will trigger after the setter signal.
- Because that second trigger may be absent, inconsistent, or strongly preload-dependent, there is no reliable per-cycle feedback signal to close a dynamic correction loop.
- As a result, a dynamic "learn each cycle" compensation model is not dependable for this hardware path.

#### Why zprobe.three_axis_probe_tlo_correction is repurposed here

- The firmware already has a stable additive correction term for probe-tool TLO storage: `zprobe.three_axis_probe_tlo_correction`.
- This value is applied when saving TLO for 3D probe tools (tool IDs in the probe range), acting as a deterministic bias correction.
- In `zprobe.use_3dtoolsetter=true` mode, this static correction becomes the practical control knob to account for spring/preload-induced Z bias when dual-trigger feedback is unavailable.
- This is intentionally explicit and operator-tunable: users determine the best value by trial cuts and repeatability checks, then keep it fixed unless hardware preload changes.

#### Maintainer note on intent and scope

- This is a controlled, low-risk compatibility strategy, not a claim that static correction is physically ideal.
- It preserves safe operation across both workflows:
  - legacy dual-trigger users (default behavior unchanged)
  - 3D toolsetter users (single-trigger stop allowed, static correction available)
- A future dynamic model should only be introduced if hardware or sensing adds a repeatable second reference event suitable for closed-loop estimation.

### Spindle direction and runtime safety

In order to gain control of the spindle the 3d toolsetter mod requires folks apply a patch line from the ext port of the controller to the fw / rev IO of the spindle controller (assuming stock spindle + controller). To make the firmware backwards compatible the following additions have been introduced to the firmware.
- Added configurable spindle direction pin definition with inversion support.
- Prevented live spindle direction reversal while running (back to back M3, M4).
- Moved direction pin setup to module load to avoid runtime config-cache access faults.
- added config.txt entries for spindle direction pin and inversion which override the EXT port functionality when defined.

## Fusion 360 CAM Compensation Considerations

### Background: Fusion 360 Compensation Options

Fusion 360 CAM offers four compensation modes for milling toolpaths: https://www.autodesk.com/products/fusion-360/blog/cutter-compensation/

| Mode | G-code Output | Offset Responsibility |
|---|---|---|
| **In Computer** | No G41/G42 | All offset applied in CAM; machine runs nominal path |
| **In Control** | G41/G42 + full tool radius in D-word | Machine applies full offset from D-word |
| **Wear** | G41/G42 + difference from nominal in D-word | CAM pre-offsets path; machine applies only the measured wear delta |
| **Inverse Wear** | G41/G42 + negative difference in D-word | Inverse of above; negative D-word value |
| **OFF** | Path is rendered on the centerline of the cutting bit with no compensation or remarks from the post processor | CAM applies no offset; machine runs nominal path |

### Current Support Status

**This branch implements "In Computer" (full offset in CAM) and "In Control" (full controller-side compensation via G41/G42) modes only.**

- **In Computer / Off:** Fully supported. CAM handles all offset geometry; the firmware receives a pre-offset toolpath and G41/G42 are not active. This is the current default and the safe fallback for all users.
- **In Control (Full Compensation):** Supported. G41/G42 activate the firmware compensation pipeline. The D-word provides the full tool radius. When a D-word is present in the gcode, it is used directly. When no D-word is present, the firmware falls back to the `TOOL_DIA` config parameter via the M491.2 measurement cycle.
- **Wear:** Not currently supported. See below.
- **Inverse Wear:** Not currently supported. No path to support exists until a tool table is available. See below.

### Fusion 360 Post Processor Guidance: Where to Place G4x and Why

This section is intended for the person editing the Fusion 360 post.
It defines exactly where `G40`, `G41`, and `G42` should be emitted in finishing toolpaths and explains the geometric reason.

#### Critical placement rules

1. Do **not** enable `G41`/`G42` during rapid positioning or during pure Z approach/retract moves.
2. Emit `G41`/`G42` immediately before the first **XY cutting lead-in** move at cutting depth.
3. Keep compensation active through the full compensated contour.
4. Emit `G40` only after a dedicated **XY lead-out** move, then perform retract/rapid moves.
5. Do not make the first compensated move an ambiguous transition (for example, compensation toggle followed immediately by non-cutting reposition logic).

#### Why this matters

The compensation pipeline is a geometric lookahead/intersection solver, not a simple per-line offset.
It computes compensated junctions from neighboring motion vectors. If compensation is toggled across very short or non-cutting transition moves, the entry/exit vectors become poorly conditioned and visible "funky" lead-in/lead-out behavior can occur even when in-path geometry is accurate.

Observed machine behavior confirms this: once `TOOL_DIA` is correct, contour body dimensions can be highly accurate while entry/exit artifacts remain if lead placement is poor.

#### Minimum lead geometry guidance for post output

- Lead-in and lead-out should be explicit XY cutting moves.
- Recommended lead length is at least the active tool radius, preferably `1.5x` to `2.0x` radius for robustness.
- For a `3.16 mm` tool (`~1.58 mm` radius), target lead lengths around `2.5 mm` to `3.5 mm`.

#### Recommended output pattern (conceptual)

```gcode
; approach and plunge without compensation
G0 X... Y...
G1 Z... F...

; enable compensation on first XY cutting lead-in
G41 D...    ; or G42 D...
G1 X... Y... F...

; compensated contour body
G1/G2/G3 ...

; dedicated XY lead-out while still compensated
G1 X... Y...
G40

; then retract/rapid
G0 Z...
G0 X... Y...
```

#### Summary for post maintainers

When editing the Fusion post, treat `G4x` emission as part of entry/exit geometry design, not just modal state toggling.
Correct placement of `G41`/`G42`/`G40` is required to preserve smooth compensated transitions at contour start/end.

### The Wear Compensation Problem: Why a Tool Table Is Required

Wear mode is operationally attractive because it produces shorter lead-in and lead-out moves — the D-word value represents only the small measured difference between the nominal programmed diameter and the actual measured diameter, rather than the full tool radius. This makes the geometry tighter and more predictable on real parts.

However, implementing Wear mode correctly requires the controller to know the **nominal (intended) tool diameter** at runtime so it can interpret the D-word delta meaningfully:

> `actual_offset = nominal_radius + D_word_value`

Without knowing the nominal diameter, the firmware cannot reconstruct the correct offset from a Wear-mode D-word alone. The D-word in Wear mode is meaningless without the baseline it refers to.

There are two paths to provide this nominal diameter to the firmware:

1. **Tool table hosted on the controller:** The controller maintains a table of tool numbers → nominal diameters. When Fusion posts a tool call (`T1 M6`), the controller looks up the nominal diameter for tool 1 and uses it as the baseline for interpreting the Wear D-word. This is the standard approach used by industrial controls (Fanuc, Haas, GRBL-HAL with tool table extensions).

2. **Read nominal diameter from gcode:** Fusion's post-processor could be configured to emit the nominal diameter as a comment or custom M-code alongside the Wear D-word. The firmware would parse it and apply it per-tool. This is a non-standard workaround and creates post-processor maintenance burden.

**Neither mechanism is currently implemented in this firmware or in the Carvera controller software.** Until one of these is available, Wear compensation cannot be enabled safely. Activating G41/G42 with a Wear-mode D-word and no nominal diameter baseline would produce incorrect offsets silently.

### The D-Word as a Simple Compensation Interface

The D-word (diameter offset register number in standard CNC; actual diameter value in this implementation) provides a straightforward path for the controller to issue compensation without requiring the CAM software to pre-bake all offset geometry into the toolpath. In this firmware:

- When a D-word is present in the gcode block activating G41/G42, it is used directly as the tool diameter for computing the offset.
- When no D-word is present, the firmware falls back to `TOOL_DIA` from `config.txt` (populated by the M491.2 certified measurement cycle).

The **compensation mode** (full vs. wear) should ultimately be a machine-level setting in `config.txt`, not determined by the CAM software alone. This allows operators to configure the machine's interpretation of incoming D-word values without requiring gcode changes.

### Petition for Tool Table Support

The community should petition for a tool table in either the firmware or the Carvera controller software. A minimal tool table would need to:

- Map tool numbers (T1, T2 … Tn) to nominal diameters.
- Be writeable from the controller UI or via M-codes (e.g., `M10 P1 R6.35` to set tool 1 nominal = 6.35 mm).
- Be readable by the compensation pipeline at G41/G42 activation time.
- Survive power cycles (stored in flash or SD card).

With a tool table in place, the path to Wear mode support becomes straightforward: at G41/G42 activation, look up the nominal diameter for the active tool number, then apply `offset = (nominal_diameter / 2) + D_word_delta`.

Until this infrastructure exists, **"In Computer" (Off) remains the safe default**, and **"In Control" (full G41/G42 with D-word or TOOL_DIA fallback) is the only controller-side compensation mode this branch supports.**
